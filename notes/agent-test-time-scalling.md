# Agent中的test-time scalling是什么

> 基于对话《测试时扩展方法》的整理笔记。

## 一句话定义

在 Agent 中，**test-time scalling（测试时扩展）**不是只多采样几段文本，而是：

**在推理阶段增加规划与环境交互计算（如更多步骤、更多工具调用、更多轨迹搜索），用算力换任务成功率。**

---

## 与普通 LLM 的区别

### 普通 LLM 的 test-time scaling
- 输入 `x`，生成输出 `y`
- 主要扩展方式：多次采样、self-consistency、rerank
- 优化目标：提高 single-shot 输出质量

### Agent 的 test-time scaling
- 输入任务后，经历 `plan -> act -> observe -> revise` 的循环
- 扩展对象从 token 级别变成了 trajectory（状态-动作序列）级别
- 优化目标：提高端到端任务成功率（例如测试通过率）

---

## Agent 场景中的典型扩展方式

1. **多路径动作搜索（Tree Search over Actions）**
   - 同一状态下展开多个 action 分支
   - 评估多条轨迹后选择更优路径

2. **增加 rollout 数（n trajectories）**
   - 例如同一需求生成多个 patch 并行验证
   - 本质是 trajectory 级别的 pass@k

3. **增加交互步数（More Steps / Longer Horizon）**
   - 给 agent 更多 debug 与修正轮次
   - 允许“先错后改”以提升最终成功率

4. **多 Agent 并行与投票（Ensemble）**
   - 生成器、批评器、验证器分工
   - 或多个 agent 并行解题后投票

5. **加深工具调用链路（Tool-use Depth）**
   - 检索 -> 阅读文件 -> 执行测试 -> 反思 -> 再检索
   - 将一次性回答变为图式推理与决策

---

## RL 视角下的理解

- LLM 常见形式：从多个候选答案里选最优 `y*`
- Agent 常见形式：从多个候选轨迹里选高回报 `τ*`

可以把 Agent 的 test-time scalling理解为：

- 对策略 `π(a_t|s_t)` 进行多次采样
- 在环境中获得回报 `R(τ)`
- 用“更多采样 + 更多交互”逼近更优轨迹

核心变化：

- **LLM：扩 token 采样空间**
- **Agent：扩状态-动作搜索空间**

---

## 代价与收益

### 收益
- 提高复杂任务成功率（修 bug、代码修复、工具链任务）
- 提升鲁棒性，减少一次性采样失败

### 代价
- 计算与时延显著上升
- 工具调用与环境执行成本高（测试、容器、I/O）
- 搜索空间快速膨胀，需要剪枝和预算控制

---

## 实践建议（面向 code-agent）

- 先设**预算上限**：最大步数、最大轨迹数、最大工具调用数
- 引入**快速验证器**：优先过滤低质量轨迹
- 采用**分层策略**：先轻量探索，再对高潜力分支加算力
- 监控**单位算力收益**：每增加一次 rollout 的通过率增益

---

## 总结

在 Agent 中，test-time scalling 的本质是：

> **扩展 trajectory 搜索，而非仅扩展文本生成。**

它是“用推理时计算换成功率”的工程化策略，尤其适合高反馈、可验证、可迭代的任务环境。
