## 一、文章主线

这篇文章把 Skills 放在一个更大的工程谱系里讨论：

- Prompt
- 结构化提示/工具调用
- Command（控制平面）
- Metadata（发现/门控）
- References（按需知识）
- Scripts（确定性执行）

核心观点：

> **Skill 不是“提示词模板”，而是可移植、可治理的过程能力包（SOP 包）。**

---

## 二、我提炼的 Skill 定义

一个 Skill 可以抽象为：

- 发现层：`name/description`（用于低成本匹配）
- 执行层：`SKILL.md`（过程与规则）
- 资源层：`references/ assets/ scripts/`（按需加载）
- 治理层：metadata + 权限策略 + 运行门控

它的重点不是“会不会回答”，而是“能不能稳定、安全、可审计地完成任务”。

---

## 三、OpenClaw 相关机制（结合实操）

文中和我当前使用体验一致的点：

1. **多目录技能加载与覆盖优先级**  
2. **load-time gating**：按 OS / binary / env / config 过滤可用技能  
3. **可用技能清单注入系统提示**（以低成本方式让模型知道“有哪些能力”）  
4. **slash commands 联动**：部分命令可绕过模型走确定性工具分发  
5. **脚本执行与工具回填**：形成 agent loop 闭环

---

## 四、两条最重要的工程原则

### 1) “确定性外包”原则

可确定的事（解析、校验、格式化、diff、批处理）优先交给脚本和工具，模型负责规划/整合，减少“纯语言推理执行”的误差。

### 2) “渐进式披露”原则

启动时只加载轻量元数据，命中后再加载正文与资源，避免上下文窗口被大文件占满。

---

## 五、安全启发（很实用）

1. 把第三方 Skills 当作潜在不可信输入，默认最小权限。  
2. 对 scripts 做供应链治理（审计、版本、可回滚）。  
3. 高风险工具（exec/browser/web）应有独立策略门控。  
4. “元数据质量”是安全边界的一部分，description 误导会导致技能误选。

---

## 六、可落地实践清单（给自己）

1. 为每个 skill 建立统一模板：目标、触发、输入输出、失败恢复。  
2. 把“是否可执行”写进 metadata，而不是靠正文提醒。  
3. 建一个 skill lint/check 清单：命名、frontmatter、路径、引用、脚本入口。  
4. 对关键技能增加最小回归测试（成功路径 + 异常路径）。

---

## 七、一句话总结

**MCP 负责“怎么连能力”，Skill 负责“怎么把能力用成流程”。**  
OpenClaw 的价值在于把这套机制平台化，但安全治理必须同步平台化。
